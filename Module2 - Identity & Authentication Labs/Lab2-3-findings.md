# TOTP Failure Root Causes Analysis

**Question:** What causes TOTP failure in an enterprise? List 3 root causes and 1 diagnostic step each.

### 1. Client Device Clock Drift (Time Synchronization Failure)
Because TOTP relies strictly on the current UNIX epoch time to generate matching codes, if the user's mobile device clock falls out of sync with the authenticating server's clock (typically beyond a 30-second window), the generated codes will be rejected.
* **Diagnostic Step:** Ask the user to navigate to a time-checking website like `time.is` on their device to check for clock drift, or have them toggle "Set time automatically" off and back on in their device settings to force an NTP sync.

### 2. Secret Key Desynchronization (Token Reset/Overwrite)
If an enterprise administrator resets the user's MFA token on the backend server, the server generates a new shared secret. If the user continues trying to use the old entry in their Authenticator app without scanning the newly provisioned QR code, authentication will fail. 
* **Diagnostic Step:** Review the enterprise identity provider's (IdP) audit logs to see if a token reset, revocation, or re-enrollment event was recently triggered for that specific user account. 

### 3. Time-Step or Hash Algorithm Mismatch
While most standard apps default to a 30-second validity period and SHA-1 hashing (per RFC 6238), some enterprise policies enforce stricter 60-second periods or stronger algorithms like SHA-256/SHA-512. If the user scans the QR code using a basic authenticator app that does not support parsing these custom parameters from the URI, it will default to 30s/SHA-1, generating permanently invalid codes.
* **Diagnostic Step:** Inspect the raw Provisioning URI (`otpauth://totp/...`) generated by the server to check the `period=` and `algorithm=` parameters, and verify if the specific authenticator app installed on the user's device fully supports those custom settings.
